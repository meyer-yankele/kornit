OBO Identity Flow Architecture

Secure Multi-Agent Identity Propagation on Azure

👤
User (David / Sarah)
Postman → simulates Teams
  • Login via Entra ID
  • Receives Token A (JWT)
  • Scope: user_impersonation
  • MFA enforced
Token A (JWT)
🧠
AIRouter — The Supervisor
Azure App Service • .NET 8 • Port 443
  • Validates Token A via Microsoft.Identity.Web
  • OBO Exchange: Token A → Token B (scoped for FakeBobData)
  • Token B stored in RAM only (InMemoryTokenCaches)
  • Creates Semantic Kernel + BobDataPlugin
  • Sends TEXT ONLY to Agent (no token)
  • Intercepts requires_action → executes plugin locally
TEXT ONLY — NO TOKEN
Token B (OBO) from RAM
🔒 Security Boundary — Token never crosses
🤖
Azure OpenAI Agent
Azure AI Foundry • GPT-4o-mini
  • Receives only text prompt
  • Analyzes intent
  • Returns requires_action
  • Function: get_hr_profile
  • Has NO token access
  • Cannot exfiltrate credentials
🗄️
FakeBobData API
Azure App Service • .NET 8
  • Validates Token B (OBO)
  • Reads upn from JWT claims
  • Returns user-specific HR data
  • David → R&D Ink Chemistry
  • Sarah → Digital Printing
  • Enforces per-user RBAC
Flow en 6 étapes
01
Authentification — L'utilisateur se connecte via Entra ID. Postman reçoit le Token A (JWT signé).
02
OBO Exchange — AIRouter échange Token A → Token B pour FakeBobData. Stocké en RAM uniquement.
03
Prompt envoyé — Seul le texte est envoyé à l'agent IA. Aucun token.
04
requires_action — L'agent IA demande l'exécution de get_hr_profile.
05
Plugin local — Semantic Kernel exécute le plugin avec Token B depuis la RAM. Appelle FakeBobData.
06
Réponse — Les données RH reviennent → l'agent formule la réponse en langage naturel.
🛡️
Zero-Trust
Token B ne quitte jamais la RAM du serveur. L'agent IA n'y a aucun accès.
🔐
Isolation d'identité
David voit ses données. Sarah voit les siennes. Prouvé par la demo live.
🔍
Traçabilité
Chaque appel porte l'identité de l'utilisateur réel. Audit complet.
